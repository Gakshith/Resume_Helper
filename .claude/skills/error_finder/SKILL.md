# Bug Checker Skill

Scans the entire project and writes all discovered bugs, errors, and issues to a `BUGS.md` file.

---

## Step 1 — Discover the Project

```bash
# Get a full picture of the project layout
find . -type f \
  ! -path '*/node_modules/*' \
  ! -path '*/.git/*' \
  ! -path '*/dist/*' \
  ! -path '*/build/*' \
  ! -path '*/__pycache__/*' \
  ! -path '*/.venv/*' \
  ! -path '*/venv/*' \
  | sort
```

Also read `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, or whatever manifest is present to understand the language/framework stack.

---

## Step 2 — Detect Language & Run Appropriate Linters/Checkers

Pick checkers based on what's in the project. Run all that apply.

### JavaScript / TypeScript
```bash
# Type errors
npx tsc --noEmit 2>&1

# Lint errors
npx eslint . --ext .js,.jsx,.ts,.tsx 2>&1

# Dependency issues
npm audit --json 2>&1
```

### Python
```bash
# Syntax + undefined names + imports
python -m py_compile $(find . -name "*.py" ! -path "*/.venv/*" ! -path "*/__pycache__/*") 2>&1

# Static analysis (install if missing)
pip install pyflakes --quiet && python -m pyflakes . 2>&1

# Optional: ruff if available
ruff check . 2>&1 || true
```

### Rust
```bash
cargo check 2>&1
cargo clippy 2>&1
```

### Go
```bash
go vet ./... 2>&1
staticcheck ./... 2>&1 || true
```

### General (all languages)
```bash
# Look for common patterns: TODO/FIXME/HACK/BUG markers
grep -rn --include="*.js" --include="*.ts" --include="*.py" --include="*.go" --include="*.rs" \
  -E "(TODO|FIXME|HACK|BUG|XXX):" . \
  ! -path '*/node_modules/*' ! -path '*/.git/*' 2>&1 || true

# Look for hardcoded secrets / credentials (naive check)
grep -rn --include="*.js" --include="*.ts" --include="*.py" --include="*.env*" \
  -iE "(password|secret|api_key|apikey|token)\s*=\s*['\"][^'\"]{4,}" . \
  ! -path '*/node_modules/*' ! -path '*/.git/*' 2>&1 || true
```

## Step 3 — Deep Static Review (Claude's own analysis)

After running automated tools, perform your own read-through of the most important source files. Look for:

- **Logic errors**: off-by-one, wrong condition, incorrect operator
- **Null / undefined dereferences**: accessing properties without guards
- **Unhandled promise rejections** (JS/TS): `async` functions without `try/catch`
- **Resource leaks**: file handles, DB connections opened but not closed
- **Incorrect error handling**: silent catches (`catch (e) {}`), swallowed exceptions
- **Race conditions**: shared mutable state, async writes without locks
- **Type mismatches**: passing wrong types, implicit coercions
- **Security issues**: SQL injection, XSS, path traversal, eval usage, unsanitized input
- **Dead code**: unreachable branches, unused variables / imports
- **Missing edge cases**: empty array, zero, null, very large input

Focus on files flagged by linters first, then highest-risk files (auth, DB, API routes, core logic).

## Step 4 — Write BUGS.md

Write the file to the **project root** at `./BUGS.md`.

### BUGS.md format

```markdown
# 🐛 BUGS Report

Generated: <ISO timestamp>
Project: <project name or directory>
Checker: Claude Bug Checker

---

## Summary

| Severity | Count |
|----------|-------|
| 🔴 Critical | N |
| 🟠 High     | N |
| 🟡 Medium   | N |
| 🔵 Low      | N |
| ℹ️ Info     | N |

---

## Issues

### 🔴 Critical

#### [CRIT-001] <Short title>
- **File**: `path/to/file.ts:42`
- **Type**: `<NullPointerException | SQLInjection | etc>`
- **Description**: Clear explanation of the bug and why it's dangerous.
- **Suggested Fix**: What should be done to fix it (code snippet if helpful).

---

### 🟠 High

#### [HIGH-001] <Short title>
...

---

### 🟡 Medium

...

---

### 🔵 Low

...

---

### ℹ️ Info / Code Quality

...

---

## Tool Output

### TypeScript / ESLint
\`\`\`
<raw linter output>
\`\`\`

### Python / Pyflakes
\`\`\`
<raw output>
\`\`\`

*(Include only tools that were run and produced output)*

---

*Generated by the Claude Bug Checker skill.*
```

### Severity Definitions

| Severity | When to use |
|----------|------------|
| 🔴 Critical | Security vulnerability, data loss risk, crash on startup, broken core feature |
| 🟠 High | Likely runtime error, incorrect business logic, unhandled exception paths |
| 🟡 Medium | Edge case bugs, performance issues, potential data corruption under load |
| 🔵 Low | Code smell, minor logic issue unlikely to trigger in practice |
| ℹ️ Info | TODO/FIXME markers, dead code, style issues, missing docs |

## Step 5 — Report to User

After writing BUGS.md:
1. Call `present_files` with the path to `BUGS.md` (if available).
2. Give a brief conversational summary: total issue count, most critical finding, what to fix first.
3. Do **not** dump the whole file into chat — just the summary.

---

## Notes

- If there are **zero issues found**, still write BUGS.md with a clean bill of health and a "No issues found" section.
- If the project is very large (>500 source files), focus on: entry points, auth/security layers, data access layer, and any files with existing linter errors. Note in BUGS.md that the scan was partial and which directories were prioritized.
- Always prefer running automated tools over manual scanning — they're faster and more reliable. Manual scanning supplements tool output.
- Append the raw tool output at the bottom of BUGS.md so the user can see what was run.
